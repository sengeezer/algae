## Why It Surfaces

This is a structured brute-force search. The question is not only how to test palindromes, but how to avoid scanning the full factor space in an arbitrary order when you only need the largest result.

## Worked Example

For `2` digits, the largest palindrome product is `9009 = 91 * 99`. For `3` digits, the same strategy scales up by searching products from the top down and returning the first palindrome that still factors into two `3`-digit numbers.

function isPalindromeInteger(value) {
  let remaining = value;
  let reversed = 0;

  while (remaining > 0) {
    reversed = reversed * 10 + (remaining % 10);
    remaining = Math.floor(remaining / 10);
  }

  return reversed === value;
}

function problem004(digits) {
  if (!Number.isInteger(digits) || digits < 1) {
    throw new RangeError('Digit count must be a positive integer');
  }

  const upper = 10 ** digits - 1;
  const lower = 10 ** (digits - 1);

  for (let product = upper * upper; product >= lower * lower; product -= 1) {
    if (!isPalindromeInteger(product)) {
      continue;
    }

    const limit = Math.ceil(Math.sqrt(product));
    for (let factor = upper; factor >= limit; factor -= 1) {
      if (product % factor === 0) {
        const partner = product / factor;
        if (partner >= lower && partner <= upper) {
          return product;
        }
      }
    }
  }

  return NaN;
}

## Follow-up Questions

- Search factor pairs directly instead of descending over every possible product.
- Explain why checking factors down to the square root is enough.
- Reuse the palindrome helper for string and integer variants.

## Common Pitfalls

- Searching in ascending order and missing the chance to stop early at the optimum.
- Failing to enforce that both factors have the requested digit count.
- Reversing the whole search space without bounding the factor check by the square root.
