## Why It Surfaces

When the objective is to minimize the worst assigned distance, crossings are harmful. Sorting both lists and pairing the `i`th mouse with the `i`th hole removes those crossings, and any alternative assignment cannot improve the largest travel time.

## Worked Example

For `mices = [4, -4, 2]` and `holes = [4, 0, 5]`, sorting gives `[-4, 2, 4]` and `[0, 4, 5]`. The travel times are `4`, `2`, and `1`, so the best possible answer is `4`.

## Reference Implementations

<ExamplePair>

```js
function assignHolesToMice(micePositions, holePositions) {
  if (micePositions.length !== holePositions.length) {
    return -1;
  }

  const sortedMice = [...micePositions].sort((left, right) => left - right);
  const sortedHoles = [...holePositions].sort((left, right) => left - right);

  let minimumTime = 0;

  for (let index = 0; index < sortedMice.length; index += 1) {
    minimumTime = Math.max(
      minimumTime,
      Math.abs(sortedMice[index] - sortedHoles[index]),
    );
  }

  return minimumTime;
}
```

## Follow-up Questions

- Return the actual mouse-to-hole assignment pairs, not only the minimized time.
- Explain the exchange argument that proves crossing pairs can be uncrossed without worsening the bottleneck.
- Adapt the function if holes can hold more than one mouse.

## Common Pitfalls

- Minimizing the total movement instead of the maximum movement.
- Sorting only one array or pairing each mouse with the nearest currently unused hole greedily.
- Mutating the input arrays unexpectedly when the caller may still need their original order.
