## Why It Surfaces

Once you can detect a cycle, interviewers often ask for the harder half: locating the exact entry node. The solution still uses fast and slow pointers, but the second phase depends on a neat distance argument rather than any new data structure.

## Worked Example

If fast and slow pointers meet inside the cycle, move one pointer back to the head and then advance both one step at a time. Their next meeting point is the node where the cycle begins.

<Callout title="Key proof idea" tone="note">
  If the non-cyclic prefix has length `L` and the cycle length is `C`, then after the first meeting the slow pointer is offset from the entry by exactly the same distance the head pointer still needs to travel.
</Callout>

## Reference Implementations

<ExamplePair>

```js
function detectCycleStart(head) {
  let slow = head;
  let fast = head;

  while (fast !== null && fast.next !== null) {
    slow = slow.next;
    fast = fast.next.next;

    if (slow === fast) {
      let entry = head;

      while (entry !== slow) {
        entry = entry.next;
        slow = slow.next;
      }

      return entry;
    }
  }

  return null;
}
```

```ts
type Node = {
  value: number;
  next: Node | null;
};

export function detectCycleStart(head: Node | null): Node | null {
  let slow = head;
  let fast = head;

  while (fast !== null && fast.next !== null) {
    slow = slow?.next ?? null;
    fast = fast.next.next;

    if (slow !== null && slow === fast) {
      let entry = head;

      while (entry !== slow) {
        entry = entry?.next ?? null;
        slow = slow.next;
      }

      return entry;
    }
  }

  return null;
}
```

## Follow-up Questions

- Prove why the second meeting occurs at the cycle entry.
- Compare this approach with a hash-set solution.
- Return the cycle length as well.

## Common Pitfalls

- Stopping after the first fast-slow meeting and returning the wrong node.
- Forgetting the null checks on `fast` and `fast.next`.
- Confusing the cycle entry with the first meeting point.
