## Why It Surfaces

This looks like a scheduling-construction problem, but the greedy core is even simpler: once the jobs are ordered in time, only the maximum number of simultaneous intervals matters. If that overlap count never exceeds two, the schedule is feasible; if it ever reaches three, no assignment can save it.

## Worked Example

For `jobs = [[1, 2], [2, 3], [4, 5]]`, the first two jobs overlap at time `2`, so the peak concurrency is exactly `2` and the schedule is feasible. For `jobs = [[1, 5], [2, 4], [2, 6], [1, 7]]`, the overlap count reaches `4`, so the answer is `false`.

## Reference Implementations

<ExamplePair>

```js
function jobSchedulingWithTwoJobsAllowedAtATime(jobs) {
  const events = [];

  for (const [start, end] of jobs) {
    events.push([start, 1]);
    events.push([end, -1]);
  }

  events.sort((left, right) => left[0] - right[0] || right[1] - left[1]);

  let activeJobs = 0;

  for (const [, delta] of events) {
    activeJobs += delta;

    if (activeJobs > 2) {
      return false;
    }
  }

  return true;
}
```

## Follow-up Questions

- Generalize the check to support an arbitrary capacity `k` instead of exactly two workers.
- Return the first time segment that makes the schedule infeasible.
- Build an explicit assignment of jobs to worker `A` or `B` when the answer is `true`.

## Common Pitfalls

- Processing end events before start events at the same time even though the problem treats touching endpoints as overlapping.
- Treating this as interval merging instead of a concurrency-bound check.
- Forgetting that one very long job can overlap many shorter non-overlapping jobs without exceeding capacity two.
