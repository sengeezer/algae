## Why It Surfaces

This is a good greedy proof problem because the move looks almost too simple: choose the largest unit fraction that still fits. The interesting part is explaining why that choice always leaves another smaller positive fraction to decompose.

## Worked Example

For `6 / 14`, the next unit fraction must be at most `1 / 3` because `1 / 2` is too large. Subtracting `1 / 3` leaves `4 / 42`, which reduces to `2 / 21`. Repeating the same rule produces `1 / 11` and then `1 / 231`, so the final representation is `1 / 3 + 1 / 11 + 1 / 231`.

## Reference Implementations

<ExamplePair>

```js
function greedyAlgorithmForEgyptianFraction(numerator, denominator) {
  if (numerator <= 0 || denominator <= 0) {
    throw new RangeError('numerator and denominator must be positive');
  }

  const terms = [];
  let currentNumerator = numerator;
  let currentDenominator = denominator;

  while (currentNumerator !== 0) {
    if (currentDenominator % currentNumerator === 0) {
      terms.push(`1/${currentDenominator / currentNumerator}`);
      break;
    }

    const nextDenominator = Math.ceil(currentDenominator / currentNumerator);
    terms.push(`1/${nextDenominator}`);
    currentNumerator = currentNumerator * nextDenominator - currentDenominator;
    currentDenominator *= nextDenominator;
  }

  return terms;
}
```

## Follow-up Questions

- Return the decomposition as reduced numerator-denominator pairs instead of strings.
- Explain why the next denominator is `ceil(denominator / numerator)`.
- Discuss how integer overflow can appear when denominators grow quickly.

## Common Pitfalls

- Choosing any smaller unit fraction instead of the largest valid one, which can increase the number of terms unnecessarily.
- Forgetting to stop when the remainder is already a unit fraction.
- Using floating-point subtraction instead of exact integer arithmetic on numerators and denominators.
