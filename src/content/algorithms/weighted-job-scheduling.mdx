## Why It Surfaces

Weighted job scheduling is what interval scheduling becomes once the objective changes from "fit as many as possible" to "maximize total value." The dynamic-programming step is to compare skipping the current job with taking it plus the best compatible profit that ends before it starts.

## Worked Example

For jobs `[(1, 3, 50), (2, 4, 10), (3, 5, 40), (3, 6, 70)]`, the best choice is `(1, 3, 50)` plus `(3, 6, 70)` for a total profit of `120`.

## Reference Implementations

<ExamplePair>

```js
function weightedJobScheduling(jobs) {
  const sortedJobs = [...jobs].sort((left, right) => left.finish - right.finish);
  const bestProfit = new Array(sortedJobs.length).fill(0);

  function findLatestCompatible(index) {
    let low = 0;
    let high = index - 1;
    let answer = -1;

    while (low <= high) {
      const mid = Math.floor((low + high) / 2);
      if (sortedJobs[mid].finish <= sortedJobs[index].start) {
        answer = mid;
        low = mid + 1;
      } else {
        high = mid - 1;
      }
    }

    return answer;
  }

  for (let index = 0; index < sortedJobs.length; index += 1) {
    const compatible = findLatestCompatible(index);
    const take = sortedJobs[index].profit + (compatible === -1 ? 0 : bestProfit[compatible]);
    const skip = index === 0 ? 0 : bestProfit[index - 1];
    bestProfit[index] = Math.max(take, skip);
  }

  return sortedJobs.length === 0 ? 0 : bestProfit[bestProfit.length - 1];
}
```

## Follow-up Questions

- Reconstruct the chosen jobs instead of only returning the profit.
- Replace the binary search with a linear predecessor scan and compare complexity.
- Explain why sorting by finish time makes the predecessor relation stable.

## Common Pitfalls

- Applying the greedy earliest-finish rule from ordinary interval scheduling and missing higher-profit combinations.
- Forgetting to add the predecessor's profit when taking a job.
- Treating touching intervals as overlapping when `finish <= nextStart` is allowed.
