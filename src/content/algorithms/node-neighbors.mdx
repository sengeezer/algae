## Why It Surfaces

Node-neighbor queries are the simplest graph primitive after adding edges. They matter because almost every graph traversal starts by asking the same local question: from this node, where can I go next?

## Worked Example

If the undirected edges are `(1, 2)`, `(2, 3)`, and `(1, 5)`, then the neighbors of `1` are `{2, 5}`. The answer only includes directly adjacent nodes, not multi-hop reachable nodes.

## Reference Implementations

<ExamplePair>

```js
function nodeNeighbors(edges, node) {
  const neighbors = new Set();

  for (const [left, right] of edges) {
    if (left === node) {
      neighbors.add(right);
    } else if (right === node) {
      neighbors.add(left);
    }
  }

  return [...neighbors];
}
```

```ts
export function nodeNeighbors(edges: Array<[number, number]>, node: number): number[] {
  const neighbors = new Set<number>();

  for (const [left, right] of edges) {
    if (left === node) {
      neighbors.add(right);
    } else if (right === node) {
      neighbors.add(left);
    }
  }

  return [...neighbors];
}
```

## Follow-up Questions

- Convert the edge list into an adjacency map for faster repeated queries.
- Explain how the logic changes for a directed graph.
- Return the neighbors in a stable sorted order.

## Common Pitfalls

- Returning nodes that are only reachable by multiple hops.
- Forgetting to deduplicate neighbors when duplicate edges exist.
- Treating a directed edge list like an undirected graph without checking the prompt.
