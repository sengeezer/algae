## Why It Surfaces

This is a classic interview constraint question. The real test is whether you understand that XOR can both combine and later separate bit patterns, not whether you can remember JavaScript destructuring syntax.

## Worked Example

Start with `10` and `20`. After the three XOR assignments, the function returns `[20, 10]`, which is the original pair in reversed order.

## Reference Implementations

<ExamplePair>

```js
function swapTwoNumbers(left, right) {
  if (!Number.isInteger(left) || !Number.isInteger(right)) {
    throw new TypeError('Inputs must be integers');
  }

  let first = left;
  let second = right;

  first ^= second;
  second ^= first;
  first ^= second;

  return [first, second];
}
```

```ts
export function swapTwoNumbers(left: number, right: number): [number, number] {
  if (!Number.isInteger(left) || !Number.isInteger(right)) {
    throw new TypeError('Inputs must be integers');
  }

  let first = left;
  let second = right;

  first ^= second;
  second ^= first;
  first ^= second;

  return [first, second];
}
```

## Follow-up Questions

- Compare the XOR version with the arithmetic-swap version and explain the overflow tradeoff.
- Rewrite the swap for array positions instead of two standalone inputs.
- Explain why a temporary variable or destructuring assignment is usually clearer in production code.

## Common Pitfalls

- Forgetting that JavaScript bitwise operators coerce values to signed 32-bit integers.
- Updating the two variables in the wrong order and losing one of the original values.
- Treating the XOR trick as automatically better than a simple temporary-variable swap.
