## Why It Surfaces

The brute-force and DP versions are straightforward, which is why interviewers like this problem: they can watch whether you notice the stronger invariant. At any point, all indices inside the current jump range are equivalent except for how far they let the next range extend.

## Worked Example

For `nums = [1, 3, 5, 8, 9, 2, 6, 7, 6, 8, 9]`, the first jump covers only index `1`. Inside that range, the farthest next reach becomes index `4`, so the second jump extends there. From that second range, the farthest reach covers the final index, so the answer is `3` jumps.

## Reference Implementations

<ExamplePair>

```js
function jumpGameMinimumJumpsToReachEnd(nums) {
  if (nums.length <= 1) {
    return 0;
  }

  if (nums[0] === 0) {
    return -1;
  }

  let jumps = 0;
  let currentReach = 0;
  let maxReach = 0;

  for (let index = 0; index < nums.length - 1; index += 1) {
    maxReach = Math.max(maxReach, index + nums[index]);

    if (index !== currentReach) {
      continue;
    }

    if (maxReach === index) {
      return -1;
    }

    jumps += 1;
    currentReach = maxReach;

    if (currentReach >= nums.length - 1) {
      return jumps;
    }
  }

  return -1;
}
```

## Follow-up Questions

- Return one optimal path of indices instead of only the minimum jump count.
- Explain the relationship between this scan and a breadth-first search over ranges.
- Contrast this minimum-jumps problem with the easier reachability-only Jump Game variant.

## Common Pitfalls

- Incrementing the jump count at every index instead of only when the current range closes.
- Missing the unreachable case where `maxReach` stops advancing.
- Confusing this problem with the version that asks only whether the last index is reachable.
