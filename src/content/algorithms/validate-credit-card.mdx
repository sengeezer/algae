## Why It Surfaces

This problem is mostly about implementing the Luhn checksum carefully. The subtle part is doubling every second digit from the right, which depends on the total length parity.

## Worked Example

For `"4532015112830366"`, the number is all digits, matches an allowed prefix, has a valid length, and its Luhn checksum ends in a multiple of `10`, so it passes.

## Reference Implementations

```js
function passesLuhn(value) {
  let sum = 0;
  let shouldDouble = false;

  for (let index = value.length - 1; index >= 0; index -= 1) {
    let digit = Number(value[index]);

    if (shouldDouble) {
      digit *= 2;
      if (digit > 9) {
        digit -= 9;
      }
    }

    sum += digit;
    shouldDouble = !shouldDouble;
  }

  return sum % 10 === 0;
}

function validateCreditCard(value) {
  if (typeof value !== 'string') {
    throw new TypeError('Value must be a string');
  }

  if (!/^\d+$/.test(value)) {
    throw new TypeError('Value must contain digits only');
  }

  if (value.length < 13 || value.length > 16) {
    return false;
  }

  if (!/^(4|5|6|34|35|37)/.test(value)) {
    return false;
  }

  return passesLuhn(value);
}
```

## Follow-up Questions

- Normalize spaces or dashes before validating the digits.
- Return a reason code instead of only a boolean.
- Support issuer-specific prefix and length rules with a configuration table.

## Common Pitfalls

- Doubling based on the left index instead of alternating from the right.
- Treating Luhn success as proof the card is real or active.
- Forgetting to reject non-digit input before the checksum pass.
