## Why It Surfaces

Greedy graph coloring is the interview version of "good enough, fast, and explainable." The important distinction is that the algorithm is only trying to produce a valid coloring in the current vertex order, not solve the NP-complete problem of minimizing colors globally.

## Worked Example

For adjacency list `[[1, 2], [0, 2, 3], [0, 1, 3], [1, 2]]`, vertex `0` gets color `0`, vertex `1` must avoid `0` so it gets `1`, vertex `2` is adjacent to both colors so it gets `2`, and vertex `3` can reuse `0`. The final coloring is `[0, 1, 2, 0]`.

## Reference Implementations

<ExamplePair>

```js
function graphColoringUsingGreedyAlgorithm(graph) {
  const vertexCount = graph.length;
  if (vertexCount === 0) {
    return { colorCount: 0, colors: [] };
  }

  const colors = new Array(vertexCount).fill(-1);
  colors[0] = 0;

  for (let vertex = 1; vertex < vertexCount; vertex += 1) {
    const unavailable = new Array(vertexCount).fill(false);

    for (const neighbor of graph[vertex]) {
      if (colors[neighbor] !== -1) {
        unavailable[colors[neighbor]] = true;
      }
    }

    let color = 0;
    while (color < unavailable.length && unavailable[color]) {
      color += 1;
    }

    colors[vertex] = color;
  }

  return {
    colorCount: Math.max(...colors) + 1,
    colors,
  };
}
```

  graph: number[][],

  colorCount: number;
  colors: number[];
} {
  const vertexCount = graph.length;
  if (vertexCount === 0) {
    return { colorCount: 0, colors: [] };

## Follow-up Questions

- Reorder vertices by descending degree and compare the color count.
- Return the color classes as groups of vertices instead of a per-vertex array.
- Explain why the algorithm never needs more than `d + 1` colors for maximum degree `d`.

## Common Pitfalls

- Claiming the greedy coloring is guaranteed to use the minimum number of colors.
- Forgetting that the vertex order can change the result materially.
- Reusing a color before checking all already-colored neighbors.
