## Why It Surfaces

This problem appears when the interviewer wants more than direct factorial counting. The useful viewpoint is to insert the current largest value into every possible position of a shorter permutation, because each insertion position contributes a predictable number of new inversions.

## Worked Example

For `n = 3` and `k = 1`, the valid permutations are `[1, 3, 2]` and `[2, 1, 3]`, so the answer is `2`. Each one contains exactly one out-of-order pair.

## Reference Implementations

<ExamplePair>

```js
function numberOfPermutationWithKInversions(size, inversions) {
  let previous = new Array(inversions + 1).fill(0);
  previous[0] = 1;

  for (let length = 1; length <= size; length += 1) {
    const current = new Array(inversions + 1).fill(0);
    let windowSum = 0;

    for (let target = 0; target <= inversions; target += 1) {
      windowSum += previous[target];
      if (target >= length) {
        windowSum -= previous[target - length];
      }
      current[target] = windowSum;
    }

    previous = current;
  }

  return previous[inversions];
}
```

## Follow-up Questions

- Return the answer modulo a large prime.
- Derive the slower O(n * k * n) transition first, then optimize it.
- Explain why inserting the maximum value contributes between `0` and `length - 1` new inversions.

## Common Pitfalls

- Mixing inversion counting with ordinary permutation generation.
- Forgetting that the largest inserted value can create only a bounded number of new inversions.
- Recomputing every transition range from scratch instead of using a sliding window.
