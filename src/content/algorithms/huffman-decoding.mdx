## Why It Surfaces

Once a Huffman tree is fixed, decoding is greedy in the sense that every next bit immediately determines the only legal subtree to enter. The moment the walk hits a leaf, that character is forced, so the traversal resets to the root and continues.

## Worked Example

Suppose the tree gives `A -> 0`, `B -> 10`, and `C -> 11`. Then the encoded bitstring `010110` is decoded by walking `0 | 10 | 11 | 0`, which reconstructs `ABCA`.

<ExamplePair>

```js
function huffmanDecoding(root, encodedBits) {
  if (root === null) {
    return '';
  }

  function isLeaf(node) {
    return node.left === null && node.right === null;
  }

  if (isLeaf(root)) {
    for (const bit of encodedBits) {
      if (bit !== '0') {
        throw new Error('Single-symbol Huffman data must use only 0 bits.');
      }
    }

    return root.character.repeat(encodedBits.length);
  }

  let result = '';
  let current = root;

  for (const bit of encodedBits) {
    if (bit !== '0' && bit !== '1') {
      throw new Error('Encoded data must contain only 0 and 1.');
    }

    current = bit === '0' ? current.left : current.right;

    if (current === null) {
      throw new Error('Encoded data does not match the Huffman tree.');
    }

    if (isLeaf(current)) {
      result += current.character;
      current = root;
    }
  }

  if (current !== root) {
    throw new Error('Encoded data ended in the middle of a code.');
  }

  return result;
}
```

## Follow-up Questions

- Decode a stream lazily without materializing the full output string first.
- Explain why prefix-free codes guarantee the walk can reset at every leaf without ambiguity.
- Extend the API so the decoder consumes the tree format emitted by your Huffman encoder.

## Common Pitfalls

- Forgetting to reset to the root after decoding one leaf.
- Accepting a bitstream that ends halfway through a root-to-leaf path.
- Ignoring the single-symbol edge case where the tree has only one leaf.
