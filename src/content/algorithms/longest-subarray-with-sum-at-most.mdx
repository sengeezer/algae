## Why It Surfaces

This is the next step after fixed-size windows: the window size changes, but the invariant still comes from a running sum. The key interview caveat is that the usual two-pointer proof only works when the values are non-negative.

## Worked Example

For `[3, 1, 2, 1, 1]` with target `4`, the longest valid window has length `3`, such as `[1, 2, 1]`. As soon as the sum exceeds `4`, move the left edge until the budget is satisfied again.

## Reference Implementations

<ExamplePair>

```js
function longestSubarrayWithSumAtMost(values, target) {
  if (values.some((value) => value < 0)) {
    throw new RangeError('Values must be non-negative for this sliding window');
  }

  let left = 0;
  let windowSum = 0;
  let best = 0;

  for (let right = 0; right < values.length; right += 1) {
    windowSum += values[right];

    while (windowSum > target && left <= right) {
      windowSum -= values[left];
      left += 1;
    }

    best = Math.max(best, right - left + 1);
  }

  return best;
}
```

## Follow-up Questions

- Return the actual window bounds instead of only the length.
- Explain why negative values break the usual shrinking rule.
- Adapt the invariant to at most `k` distinct items or at most `k` zeros.

## Common Pitfalls

- Applying the simple two-pointer version to arrays with negative numbers.
- Recording the answer before shrinking an invalid window.
- Forgetting that the result can be `0` when even a single value exceeds a negative target.
