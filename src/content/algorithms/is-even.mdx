## Why It Surfaces

Parity is one of the most common branching conditions in interview math. This helper is tiny, but it shows whether you validate integer inputs and whether you understand why bitwise parity checks only make sense for integer values.

## Worked Example

`14` is even because `14 % 2` is `0`. `17` is not even because its remainder modulo `2` is `1`.

## Reference Implementations

<ExamplePair>

```js
function isEven(value) {
  return Number.isInteger(value) && value % 2 === 0;
}

function isEvenBitwise(value) {
  return Number.isInteger(value) && (value & 1) === 0;
}
```

```ts
export function isEven(value: number): boolean {
  return Number.isInteger(value) && value % 2 === 0;
}

export function isEvenBitwise(value: number): boolean {
  return Number.isInteger(value) && (value & 1) === 0;
}
```

</ExamplePair>

## Follow-up Questions

- Explain why bitwise parity checks are only appropriate for integer inputs.
- Compare the readability of modulo and bitwise implementations.
- Use parity to simplify a larger recurrence or dynamic-programming transition.

## Common Pitfalls

- Treating non-integer values as valid even numbers.
- Forgetting that bitwise operators coerce values into 32-bit integers.
- Reimplementing divisibility logic when `% 2` already answers the question.
