## Why It Surfaces

Softmax is simple mathematically, but the real implementation detail is stability. Good answers explain why subtracting the maximum score preserves the output distribution while reducing overflow risk.

## Worked Example

For scores `[1, 2, 3]`, first shift by the maximum to get `[-2, -1, 0]`, exponentiate those values, and then divide each by their sum. The resulting probabilities are positive and sum to `1`.

## Reference Implementations

<ExamplePair>

```js
function softmax(values) {
  if (!Array.isArray(values) || values.length === 0) {
    throw new TypeError('Expected a non-empty array');
  }

  if (!values.every((value) => typeof value === 'number' && Number.isFinite(value))) {
    throw new TypeError('Values must be finite numbers');
  }

  const maxValue = Math.max(...values);
  const exponentials = values.map((value) => Math.exp(value - maxValue));
  const total = exponentials.reduce((sum, value) => sum + value, 0);
  return exponentials.map((value) => value / total);
}
```

## Follow-up Questions

- Compute the log-softmax instead and explain why it is useful.
- Compare softmax with min-max normalization.
- Show why subtracting the maximum score does not change the final probabilities.

## Common Pitfalls

- Exponentiating large raw scores directly and risking overflow.
- Forgetting that the output probabilities must sum to `1`.
- Accepting empty arrays or non-finite values without defining the behavior.
