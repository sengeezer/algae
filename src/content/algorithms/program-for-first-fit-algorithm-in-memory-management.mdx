## Why It Surfaces

First fit is the simplest nontrivial placement heuristic: every decision is local, irreversible, and changes the future search space by shrinking one block. That makes it a common interview vehicle for explaining how a greedy policy can be fast and easy to implement while still causing avoidable fragmentation later.

## Worked Example

For `blockSizes = [100, 500, 200, 300, 600]` and `processSizes = [212, 417, 112, 426]`, first fit places `212` into block `2`, `417` into block `5`, `112` into the remaining space of block `2`, and cannot place `426`. The resulting allocation is `[2, 5, 2, -1]` using 1-based block numbers.

## Reference Implementations

<ExamplePair>

```js
function programForFirstFitAlgorithmInMemoryManagement(blockSizes, processSizes) {
  const remainingBlocks = [...blockSizes];
  const allocation = Array(processSizes.length).fill(-1);

  for (let processIndex = 0; processIndex < processSizes.length; processIndex += 1) {
    for (let blockIndex = 0; blockIndex < remainingBlocks.length; blockIndex += 1) {
      if (remainingBlocks[blockIndex] < processSizes[processIndex]) {
        continue;
      }

      allocation[processIndex] = blockIndex + 1;
      remainingBlocks[blockIndex] -= processSizes[processIndex];
      break;
    }
  }

  return allocation;
}
```

## Follow-up Questions

- Return the final remaining block capacities along with the process allocations.
- Compare the fragmentation pattern of first fit against best fit on the same workload.
- Adapt the routine if a block may hold at most one process instead of shrinking after allocation.

## Common Pitfalls

- Forgetting to reduce the remaining capacity of a block after allocating a process.
- Returning zero-based block indices when the worked examples use 1-based numbering.
- Continuing the scan after a successful first-fit placement instead of stopping immediately.
