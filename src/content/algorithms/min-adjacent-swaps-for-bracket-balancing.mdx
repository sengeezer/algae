## Why It Surfaces

This problem looks like it needs explicit swap simulation, but the greedy invariant is cheaper: when there are more `]` than `[` so far, the next `[` must cross every unmatched `]` before it. That imbalance count is exactly the number of swaps contributed by that character.

## Worked Example

For `s = "[]][]["`, the third character creates an imbalance of one unmatched `]`. The next `[` repairs that imbalance for one swap, and the final `[` repairs another imbalance for one more swap, so the minimum total is `2`.

<ExamplePair>

```js
function minAdjacentSwapsForBracketBalancing(text) {
  let openingCount = 0;
  let closingCount = 0;

  for (const char of text) {
    if (char === '[') {
      openingCount += 1;
    } else if (char === ']') {
      closingCount += 1;
    }
  }

  if (openingCount !== closingCount) {
    return -1;
  }

  openingCount = 0;
  closingCount = 0;

  let imbalance = 0;
  let swaps = 0;

  for (const char of text) {
    if (char === '[') {
      openingCount += 1;

      if (imbalance > 0) {
        swaps += imbalance;
        imbalance -= 1;
      }
    } else {
      closingCount += 1;
      imbalance = closingCount - openingCount;
    }
  }

  return swaps;
}
```

## Follow-up Questions

- Compare this O(1)-space imbalance scan with the index-tracking version that stores every `[` position.
- Adapt the routine if the input may contain non-bracket characters that should be ignored or rejected.
- Explain why arbitrary swaps would change the cost model completely.

## Common Pitfalls

- Simulating each adjacent swap explicitly and turning a linear scan into quadratic work.
- Forgetting that unequal counts of `[` and `]` make balancing impossible.
- Counting future swaps when the current scan has not yet accumulated an imbalance.
