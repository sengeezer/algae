## Why It Surfaces

Email validation questions are really about scope control. The right answer is usually a pragmatic format screen, plus a clear explanation that regex alone does not prove an address can actually receive mail.

## Worked Example

`"algae.study@example.com"` passes a basic validation because it has a nonempty local part, a single `@`, and a domain containing a dot-separated suffix.

## Reference Implementations

<ExamplePair>

```js
function validateEmail(value) {
  if (typeof value !== 'string' || value.length === 0) {
    throw new TypeError('Value must be a nonempty string');
  }

  return /^[^\s@]+@[^\s@]+\.[^\s@]+$/.test(value);
}
```

```ts
export function validateEmail(value: string): boolean {
  if (value.length === 0) {
    throw new TypeError('Value must be a nonempty string');
  }

  return /^[^\s@]+@[^\s@]+\.[^\s@]+$/.test(value);
}
```

</ExamplePair>

## Follow-up Questions

- Normalize surrounding whitespace before validating.
- Distinguish between format validity and deliverability.
- Replace the simple regex with a stricter library-backed validator in production code.

## Common Pitfalls

- Claiming a regex proves an email address is real.
- Forgetting to reject empty strings.
- Overcomplicating the pattern when the prompt only needs a basic format check.
