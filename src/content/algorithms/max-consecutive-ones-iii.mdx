## Why It Surfaces

This is a strong interview window problem because the invariant is easy to state and easy to violate. The current segment is valid only while the number of zeros stays within the allowed flip budget.

## Worked Example

For `nums = [1, 1, 0, 0, 1, 1, 1]` and `k = 1`, the window can contain one zero but must shrink as soon as a second zero enters. That grow-then-shrink pattern is the whole solution.

<Callout title="Window rule" tone="tip">
  Count zeros, not ones. The budget is expressed in terms of the bad values you are allowed to absorb.
</Callout>

## Reference Implementations

<ExamplePair>

```js
function longestOnes(nums, flips) {
  let left = 0;
  let zeroCount = 0;
  let best = 0;

  for (let right = 0; right < nums.length; right += 1) {
    if (nums[right] === 0) {
      zeroCount += 1;
    }

    while (zeroCount > flips) {
      if (nums[left] === 0) {
        zeroCount -= 1;
      }

      left += 1;
    }

    best = Math.max(best, right - left + 1);
  }

  return best;
}
```

## Follow-up Questions

- Return the actual best window bounds, not just its length.
- Generalize this to strings with at most `k` replacements.
- Explain why each pointer still moves at most `n` times.

## Common Pitfalls

- Counting the wrong value and shrinking on the wrong condition.
- Forgetting to shrink until the window is valid again.
- Recomputing the number of zeros from scratch after each move.
