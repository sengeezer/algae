## Why It Surfaces

This problem looks like cryptography, but the interview core is really reverse dependency recovery. Each ciphertext digit is the XOR of a suffix of plaintext digits, so the rightmost side contains enough information to peel the plaintext back one hex digit at a time.

## Worked Example

The ciphertext `a1d0a1d` decrypts to `abcd`. The last ciphertext digit `d` is already the last plaintext digit, then each step to the left XORs with the suffix recovered so far to reveal the next plaintext digit.

## Reference Implementations

<ExamplePair>

```js
function xorEncryptionByShiftingPlaintext(ciphertext) {
  if (typeof ciphertext !== 'string' || !/^[0-9a-fA-F]+$/.test(ciphertext)) {
    throw new TypeError('Ciphertext must be a non-empty hexadecimal string');
  }

  const normalized = ciphertext.toLowerCase();
  if (normalized.length % 2 === 0) {
    throw new RangeError('Ciphertext length must be odd');
  }

  let rollingXor = 0;
  let plaintext = '';

  for (let index = normalized.length - 1; index >= Math.floor(normalized.length / 2); index -= 1) {
    const cipherDigit = Number.parseInt(normalized[index], 16);
    const plainDigit = rollingXor ^ cipherDigit;

    plaintext = plainDigit.toString(16) + plaintext;
    rollingXor ^= plainDigit;
  }

  return plaintext;
}
```

## Follow-up Questions

- Add the forward encryption routine and prove why the decryption loop is its inverse.
- Adapt the solution if the alphabet were bytes instead of hexadecimal digits.
- Stream the plaintext out without building the full answer in memory first.

## Common Pitfalls

- Treating the input as arbitrary text instead of hexadecimal digits.
- Scanning from the left even though the right edge is the only side that directly exposes plaintext digits.
- Forgetting that a valid shifted-self-XOR ciphertext has odd length.
