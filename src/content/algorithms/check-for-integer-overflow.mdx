## Why It Surfaces

This question is about respecting fixed-width arithmetic rules instead of relying on a language runtime to save you. The core observation is that overflow during addition only happens when both inputs have the same sign and the mathematical sum moves past the signed boundary.

## Worked Example

Under signed 32-bit rules, `1000000000 + 1000000000` is safe and returns `2000000000`, but `-2000000000 + -500000000` falls below `-2147483648`, so the function should return `-1` to signal overflow.

## Reference Implementations

<ExamplePair>

```js
function checkForIntegerOverflow(left, right) {
  if (!Number.isInteger(left) || !Number.isInteger(right)) {
    throw new TypeError('Inputs must be integers');
  }

  const minInt = -(2 ** 31);
  const maxInt = 2 ** 31 - 1;

  if (left < minInt || left > maxInt || right < minInt || right > maxInt) {
    throw new RangeError('Inputs must fit in a signed 32-bit integer');
  }

  if ((left >= 0 && right >= 0 && left > maxInt - right) || (left < 0 && right < 0 && left < minInt - right)) {
    return -1;
  }

  return left + right;
}
```

## Common Pitfalls

- Forgetting that JavaScript `number` does not overflow at 32-bit integer limits by itself.
- Checking only positive overflow and missing the negative case.
- Accepting inputs that are already outside the modeled integer range.
