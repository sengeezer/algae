## Why It Surfaces

This is the moment where gcd stops being just a divisor question and becomes a construction tool. Once you can return the coefficients as well, modular inverses and linear equations fall out naturally.

## Worked Example

For `30` and `18`, the gcd is `6`, and one valid coefficient pair is `(-1, 2)` because `30 * (-1) + 18 * 2 = 6`.

<Callout title="What changes" tone="note">
  The remainder updates are the same as ordinary Euclid. The only extra work is carrying two coefficient sequences alongside the remainder sequence.
</Callout>

## Reference Implementations

<ExamplePair>

```js
function extendedEuclideanGcd(a, b) {
  if (!Number.isInteger(a) || !Number.isInteger(b) || a < 1 || b < 1) {
    throw new TypeError('Inputs must be positive integers');
  }

  let oldR = a;
  let r = b;
  let oldS = 1;
  let s = 0;
  let oldT = 0;
  let t = 1;

  while (r !== 0) {
    const quotient = Math.floor(oldR / r);
    [oldR, r] = [r, oldR - quotient * r];
    [oldS, s] = [s, oldS - quotient * s];
    [oldT, t] = [t, oldT - quotient * t];
  }

  return [oldR, oldS, oldT];
}
```

## Common Pitfalls

- Returning only the gcd and losing the whole point of the extended version.
- Mixing up the coefficient order after swapping remainder states.
- Forgetting that modular inverses require the gcd to be `1`.
