## Why It Surfaces

This is a lightweight validation problem, but it is useful because it turns naming rules into an explicit predicate instead of a hand-wavy style judgment.

## Worked Example

`"userName"` passes because it starts with a lowercase letter and contains only letters afterward. `"UserName"` fails because PascalCase starts with an uppercase letter instead.

## Reference Implementations

<ExamplePair>

```js
function checkCamelCase(value) {
  if (typeof value !== 'string') {
    throw new TypeError('Value must be a string');
  }

  return /^[a-z][A-Za-z]*$/.test(value);
}
```

```ts
export function checkCamelCase(value: string): boolean {
  return /^[a-z][A-Za-z]*$/.test(value);
}
```

</ExamplePair>

## Follow-up Questions

- Convert snake_case or kebab-case into camelCase.
- Decide whether digits should be accepted and update the predicate accordingly.
- Differentiate lowerCamelCase from PascalCase in one shared classifier.

## Common Pitfalls

- Treating PascalCase as camelCase.
- Forgetting to reject underscores or hyphens.
- Hard-coding one style rule when the prompt may allow digits or acronyms.
