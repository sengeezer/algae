## Why It Surfaces

This question is usually about understanding the structure of a UUID string, not about cryptography. The interesting part is how the template mixes random hex digits with fixed version and variant bits.

## Worked Example

A generated result might look like `"c1e8db4a-53c6-4a5f-9d02-0d3e701c3b6a"`. The third group starts with `4`, and the first digit of the fourth group stays in the UUID variant range `8` through `b`.

## Reference Implementations

<ExamplePair>

```js
function generateGuid() {
  let timestamp = Date.now();

  return 'xxxxxxxx-xxxx-4xxx-yxxx-xxxxxxxxxxxx'.replace(/[xy]/g, (token) => {
    const random = (timestamp + Math.random() * 16) % 16 | 0;
    timestamp = Math.floor(timestamp / 16);

    const value = token === 'x' ? random : (random & 0x3) | 0x8;

    return value.toString(16);
  });
}
```

## Follow-up Questions

- Replace `Math.random()` with `crypto.getRandomValues()` or `crypto.randomUUID()`.
- Explain why the variant nibble must stay in the `8` to `b` range.
- Generate uppercase or brace-wrapped GUID strings for specific APIs.

## Common Pitfalls

- Treating `Math.random()` output as cryptographically secure.
- Forgetting to pin the version nibble to `4`.
- Using a loose mask that allows invalid UUID variant values.
