## Why It Surfaces

Perfect-square checks usually show up as a small subroutine inside a larger problem. The main interview question is whether you can make the root test robust enough to avoid easy floating-point mistakes.

## Worked Example

`49` is a perfect square because `Math.sqrt(49)` gives `7`, and `7 * 7 = 49`. `50` is not, because the closest integer root is `7`, and `7 * 7 = 49`, not `50`.

## Reference Implementations

<ExamplePair>

```js
function perfectSquare(value) {
  return Number.isFinite(value) && value >= 0 && Math.round(Math.sqrt(value)) ** 2 === value;
}
```

```ts
export function perfectSquare(value: number): boolean {
  return Number.isFinite(value) && value >= 0 && Math.round(Math.sqrt(value)) ** 2 === value;
}
```

</ExamplePair>

## Follow-up Questions

- Replace the built-in root with a binary-search integer square root.
- Explain why rounding is safer than truncating near floating-point boundaries.
- Extend the check to perfect cubes or other perfect powers.

## Common Pitfalls

- Forgetting that negative values are not perfect squares over the real numbers.
- Squaring a truncated root and misclassifying values because of floating-point noise.
- Assuming the approach remains exact for arbitrarily large integers outside the safe numeric range.
