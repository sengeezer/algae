## Why It Surfaces

This problem is small, but it tests whether you can be exact with string formats and edge cases. Midnight and noon are the usual failure points.

## Worked Example

`07:05:45PM` becomes `19:05:45`, while `12:00:00AM` becomes `00:00:00`. The minutes and seconds never change; only the hour and suffix drive the normalization.

## Reference Implementations

<ExamplePair>

```js
const TIME_PATTERN = /^(0[1-9]|1[0-2]):([0-5]\d):([0-5]\d)(AM|PM)$/;

function railwayTimeConversion(input) {
  if (typeof input !== 'string') {
    throw new TypeError('Expected a time string');
  }

  const match = TIME_PATTERN.exec(input);
  if (!match) {
    throw new RangeError('Time must use hh:mm:ssAM or hh:mm:ssPM format');
  }

  let hour = Number(match[1]);
  const minute = match[2];
  const second = match[3];
  const meridiem = match[4];

  if (meridiem === 'AM') {
    hour = hour === 12 ? 0 : hour;
  } else {
    hour = hour === 12 ? 12 : hour + 12;
  }

  return `${String(hour).padStart(2, '0')}:${minute}:${second}`;
}
```

## Follow-up Questions

- Accept inputs without seconds and normalize them to `hh:mm:00`.
- Explain why 24-hour strings sort correctly as plain text.
- Parse the result into a structured `{ hour, minute, second }` object.

## Common Pitfalls

- Turning `12:xx:xxAM` into `12:xx:xx` instead of `00:xx:xx`.
- Adding twelve hours to `12:xx:xxPM`, which should stay `12`.
- Accepting malformed strings like `7:5:45PM` when the prompt expects fixed-width input.
