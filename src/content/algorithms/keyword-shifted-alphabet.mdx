## Why It Surfaces

This is really a table-construction problem. Once you build the keyed alphabet correctly, encryption and decryption reduce to indexed translation while preserving case and non-alphabetic characters.

## Worked Example

With keyword `"keyword"`, the cipher alphabet begins `K E Y W O R D A B C ...`. Plaintext letters are translated position by position from the normal alphabet into that keyed sequence.

## Reference Implementations

<ExamplePair>

```js
function keywordShiftedAlphabet(keyword, message, mode = 'encrypt') {
  if (typeof keyword !== 'string' || typeof message !== 'string') {
    throw new TypeError('Keyword and message must be strings');
  }

  const normalizedKeyword = keyword.toLowerCase();
  if (!/^[a-z]+$/.test(normalizedKeyword)) {
    throw new Error('Keyword must contain only letters');
  }

  if (new Set(normalizedKeyword).size !== normalizedKeyword.length) {
    throw new Error('Keyword must not repeat letters');
  }

  const alphabet = 'abcdefghijklmnopqrstuvwxyz';
  const keyedAlphabet = normalizedKeyword + [...alphabet]
    .filter((char) => !normalizedKeyword.includes(char))
    .join('');
  const source = mode === 'encrypt' ? alphabet : keyedAlphabet;
  const target = mode === 'encrypt' ? keyedAlphabet : alphabet;

  return message.replace(/[a-z]/gi, (char) => {
    const lower = char.toLowerCase();
    const mapped = target[source.indexOf(lower)];
    return /[A-Z]/.test(char) ? mapped.toUpperCase() : mapped;
  });
}
```

## Follow-up Questions

- Normalize repeated keyword letters instead of rejecting them.
- Show the translation table explicitly for debugging.
- Compare keyword substitution with Caesar and Vigenere ciphers.

## Common Pitfalls

- Forgetting to reject or deduplicate repeated keyword letters.
- Replacing non-letter characters that should pass through untouched.
- Mixing up the source and target alphabets during decryption.
