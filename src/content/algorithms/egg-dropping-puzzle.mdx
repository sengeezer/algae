## Why It Surfaces

This is the classic worst-case experiment-design problem. The usual floor-by-floor recurrence is easy to describe but too slow, so the real interview move is to switch perspective and ask how many floors can already be resolved with a certain number of moves and eggs.

## Worked Example

For `eggs = 2` and `floors = 10`, the answer is `4`. With four moves and two eggs, the optimized DP can distinguish up to ten floors in the worst case, so that is the first move budget that guarantees certainty.

## Reference Implementations

<ExamplePair>

```js
function eggDroppingPuzzle(eggs, floors) {
  if (!Number.isInteger(eggs) || !Number.isInteger(floors) || eggs < 0 || floors < 0) {
    throw new RangeError('eggs and floors must be non-negative integers');
  }

  if (eggs === 0 || floors <= 1) {
    return floors;
  }

  const dp = new Array(eggs + 1).fill(0);
  let moves = 0;

  while (dp[eggs] < floors) {
    moves += 1;

    for (let egg = eggs; egg >= 1; egg -= 1) {
      dp[egg] = dp[egg] + dp[egg - 1] + 1;
    }
  }

  return moves;
}
```

## Follow-up Questions

- Derive why `dp[egg] = dp[egg] + dp[egg - 1] + 1` matches the break and survive branches.
- Compare this move-based DP with the older O(eggs * floors^2) floor-scan recurrence.
- Return a concrete dropping schedule for the two-egg case.

## Common Pitfalls

- Minimizing the average case when the prompt asks for the worst case.
- Updating the compressed `dp` array left to right instead of right to left.
- Returning the number of reachable floors instead of the move count that achieves them.
