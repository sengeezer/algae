## Why It Surfaces

Alpha numerical sort matters whenever user-facing order should match how people read labels rather than how machines compare raw characters. The classic failure case is that plain lexicographic sorting places `z11` before `z2`.

## Worked Example

Sorting `['file1', 'file10', 'file2']` lexicographically yields `['file1', 'file10', 'file2']`, but natural ordering yields `['file1', 'file2', 'file10']` because `10` and `2` are treated as whole numbers.

## Reference Implementations

<ExamplePair>

```js
function alphaNumericalSort(left, right) {
  return left.localeCompare(right, undefined, {
    numeric: true,
    sensitivity: 'base',
  });
}

function naturalSort(values) {
  if (!Array.isArray(values)) {
    throw new TypeError('Values must be an array of strings');
  }

  return [...values].sort(alphaNumericalSort);
}
```

```ts
export function alphaNumericalSort(left: string, right: string): number {
  return left.localeCompare(right, undefined, {
    numeric: true,
    sensitivity: 'base',
  });
}

export function naturalSort(values: string[]): string[] {
  return [...values].sort(alphaNumericalSort);
}
```

## Follow-up Questions

- Explain why plain lexicographic comparison fails on embedded numbers.
- Make the comparator case-sensitive or locale-specific.
- Extend the approach to version strings with multiple numeric segments.

## Common Pitfalls

- Treating the problem like ordinary alphabetical sorting.
- Forgetting that the comparator, not the sorting engine, is the main algorithmic decision.
- Assuming all locales and runtimes behave identically without checking the `localeCompare` options.
