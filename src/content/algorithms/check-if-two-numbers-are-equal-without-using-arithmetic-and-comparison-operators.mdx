## Why It Surfaces

This is another operator-ban puzzle whose real purpose is to test whether you know a small XOR identity. Once you see that equal numbers have no differing bits, the whole problem reduces to checking whether the XOR result is zero.

## Worked Example

`10` and `10` are equal, so `10 ^ 10` becomes `0` and the function returns `true`. For `10` and `20`, the XOR is non-zero, so the answer is `false`.

## Reference Implementations

<ExamplePair>

```js
function checkIfTwoNumbersAreEqualWithoutUsingArithmeticAndComparisonOperators(left, right) {
  if (!Number.isInteger(left) || !Number.isInteger(right)) {
    throw new TypeError('Inputs must be integers');
  }

  return !(left ^ right);
}
```

```ts
export function checkIfTwoNumbersAreEqualWithoutUsingArithmeticAndComparisonOperators(
  left: number,
  right: number,
): boolean {
  if (!Number.isInteger(left) || !Number.isInteger(right)) {
    throw new TypeError('Inputs must be integers');
  }

  return !(left ^ right);
}
```

## Follow-up Questions

- Rework the answer without using XOR either.
- Explain why the same idea works for any fixed-width integer representation.
- Contrast this with string-based equality, where the inputs are not yet parsed as numbers.

## Common Pitfalls

- Using a regular equality operator even though the prompt bans it.
- Forgetting that bitwise operators in JavaScript coerce to 32-bit signed integers.
- Overcomplicating a constant-time identity with a full bit-by-bit scan.
