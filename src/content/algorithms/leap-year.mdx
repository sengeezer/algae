## Why It Surfaces

Leap-year checks are small, but they reliably expose whether you remember the century exception instead of stopping at divisibility by `4`.

## Worked Example

`2024` is a leap year because it is divisible by `4` and not by `100`. `1900` is not a leap year because century years must also be divisible by `400`. `2000` is a leap year because it passes that `400` rule.

## Reference Implementations

<ExamplePair>

```js
function isLeapYear(year) {
  if (!Number.isInteger(year)) {
    throw new TypeError('Year must be an integer');
  }

  return year % 400 === 0 || (year % 100 !== 0 && year % 4 === 0);
}
```

```ts
export function isLeapYear(year: number): boolean {
  if (!Number.isInteger(year)) {
    throw new TypeError('Year must be an integer');
  }

  return year % 400 === 0 || (year % 100 !== 0 && year % 4 === 0);
}
```

## Follow-up Questions

- Use this helper inside date parsing or day-of-week calculations.
- Explain why the Gregorian rules need both the `100` and `400` exceptions.
- Compare this direct rule set with a lookup-table approach for a bounded year range.

## Common Pitfalls

- Treating every year divisible by `4` as a leap year.
- Forgetting that `2000` is a leap year even though `1900` is not.
- Accepting non-integer year inputs without defining the behavior.
