## Why It Surfaces

This is a count-maximization greedy problem. Once each customer request is reduced to one total demand, spending inventory on any heavier request before a lighter one can only reduce how many customers you can still satisfy.

## Worked Example

If packet sizes are `a = 1` and `b = 1`, total stock is `5`, and customer demands become `[2, 5, 8, 10, 1]`, sort those demands to `1, 2, 5, 8, 10`. Serving the customers with demand `1` and `2` uses only `3` units total, so two customers are satisfied before the remaining stock becomes too small for the next request.

## Reference Implementations

<ExamplePair>

```js
function maximumNumberOfCustomersThatCanBeSatisfiedWithGivenQuantity(
  packetASize,
  packetBSize,
  totalQuantity,
  requests,
) {
  const rankedCustomers = requests
    .map(([countA, countB], index) => ({
      customerIndex: index + 1,
      demand: packetASize * countA + packetBSize * countB,
    }))
    .sort((left, right) => left.demand - right.demand || left.customerIndex - right.customerIndex);

  const customerIndices = [];
  let remaining = totalQuantity;

  for (const customer of rankedCustomers) {
    if (customer.demand > remaining) {
      break;
    }

    remaining -= customer.demand;
    customerIndices.push(customer.customerIndex);
  }

  return {
    count: customerIndices.length,
    customerIndices,
  };
}
```

## Follow-up Questions

- Return the satisfied customer indices in original order instead of demand order.
- Explain why the greedy rule changes if customers have different priorities or profit values.
- Adapt the solution if packet inventory is split into separate `a` and `b` stock counts instead of one total quantity.

## Common Pitfalls

- Optimizing for the largest fulfilled demand instead of the largest number of fulfilled customers.
- Forgetting to collapse each two-packet request into one comparable total demand.
- Continuing after the first unaffordable customer even though all later sorted demands are at least as large.
