## Why It Surfaces

Temperature conversion is a good example of how normalization beats pairwise duplication. The upstream source exposes many direct functions; locally, it is cleaner to convert into one canonical scale and back out.

## Worked Example

`25` degrees Celsius becomes `77` degrees Fahrenheit. In a normalized implementation, the value first becomes Kelvin and then converts from Kelvin to the requested target scale.

## Reference Implementations

<ExamplePair>

```js
const toKelvin = {
  celsius: (value) => value + 273.15,
  fahrenheit: (value) => ((value - 32) * 5) / 9 + 273.15,
  kelvin: (value) => value,
  rankine: (value) => (value * 5) / 9,
  reaumur: (value) => value * 1.25 + 273.15,
};

const fromKelvin = {
  celsius: (value) => value - 273.15,
  fahrenheit: (value) => ((value - 273.15) * 9) / 5 + 32,
  kelvin: (value) => value,
  rankine: (value) => (value * 9) / 5,
  reaumur: (value) => (value - 273.15) * 0.8,
};

function temperatureConversion(value, fromUnit, toUnit) {
  if (!Number.isFinite(value)) {
    throw new TypeError('Temperature must be a finite number');
  }

  if (!(fromUnit in toKelvin) || !(toUnit in fromKelvin)) {
    throw new RangeError('Unsupported temperature unit');
  }

  const kelvin = toKelvin[fromUnit](value);
  return fromKelvin[toUnit](kelvin);
}
```

## Common Pitfalls

- Writing one function per pair of units and creating unnecessary duplication.
- Forgetting that some scales have both multiplicative and additive offsets.
- Rounding too early and losing precision across chained conversions.
