## Why It Surfaces

This is the next step after slope. Instead of representing a line as rise over run, you convert it into a reusable equation that works for horizontal, vertical, and slanted lines with the same algebraic form.

## Worked Example

For points `[3, 2]` and `[2, 6]`, we get `a = 6 - 2 = 4`, `b = 3 - 2 = 1`, and `c = 4 * 3 + 1 * 2 = 14`, so the line is `4x + 1y = 14`.

## Reference Implementations

<ExamplePair>

```js
function programToFindLinePassingThrough2Points(first, second) {
  if (first[0] === second[0] && first[1] === second[1]) {
    throw new RangeError("Distinct points are required");
  }

  let a = second[1] - first[1];
  let b = first[0] - second[0];
  let c = a * first[0] + b * first[1];

  if (a < 0 || (a === 0 && b < 0)) {
    a = -a;
    b = -b;
    c = -c;
  }

  return [a, b, c];
}
```

## Follow-up Questions

- Why does standard form handle vertical lines more cleanly than slope-intercept form?
- How would you normalize the coefficients so equivalent lines always get the same triple?
- Why do identical input points fail to define a unique line?

## Common Pitfalls

- Using the same point twice and pretending it still defines a line.
- Returning slope-intercept form even when the line is vertical.
- Forgetting to normalize sign, which makes equivalent lines harder to compare.
