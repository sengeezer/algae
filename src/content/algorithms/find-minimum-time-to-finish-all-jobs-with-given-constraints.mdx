## Why It Surfaces

The greedy step only answers a yes-or-no question: if no assignee may exceed a candidate workload limit, how many contiguous partitions are needed? Once a limit works, every larger limit also works, which makes the full problem a binary search over feasible maximum workloads.

## Worked Example

For `job = [10, 7, 8, 12, 6, 8]`, `k = 4`, and `t = 5`, the minimum feasible workload per assignee is `15`. One optimal split is `[10]`, `[7, 8]`, `[12]`, and `[6, 8]`, so the final minimum time is `15 * 5 = 75`.

## Reference Implementations

<ExamplePair>

```js
function findMinimumTimeToFinishAllJobsWithGivenConstraints(
  jobs,
  assigneeCount,
  unitTime,
) {
  if (jobs.length === 0 || assigneeCount <= 0) {
    return 0;
  }

  let low = Math.max(...jobs);
  let high = jobs.reduce((sum, value) => sum + value, 0);
  let answer = high;

  function canFinishWithin(limit) {
    let usedAssignees = 1;
    let currentLoad = 0;

    for (const job of jobs) {
      if (currentLoad + job <= limit) {
        currentLoad += job;
        continue;
      }

      usedAssignees += 1;
      currentLoad = job;

      if (usedAssignees > assigneeCount) {
        return false;
      }
    }

    return true;
  }

  while (low <= high) {
    const mid = low + Math.floor((high - low) / 2);

    if (canFinishWithin(mid)) {
      answer = mid;
      high = mid - 1;
    } else {
      low = mid + 1;
    }
  }

  return answer * unitTime;
}
```

## Follow-up Questions

- Return the actual contiguous partitions, not only the minimum time.
- Explain why greedy packing is valid for a fixed workload limit.
- Compare this with `Allocate Minimum Pages` or other split-array minimax variants.

## Common Pitfalls

- Searching the answer over time units directly without separating the per-assignee workload from the final multiplier `t`.
- Forgetting that each assignee must receive a contiguous block.
- Starting binary search below the largest single job, which can never be feasible.
