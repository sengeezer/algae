## Why It Surfaces

This problem is a good graph-DP checkpoint because plain traversal only tells you whether a destination is reachable, not how many fixed-length walks get there. The exact edge count turns path length into part of the state.

## Worked Example

If `0 -> 1 -> 3` and `0 -> 2 -> 3` are both valid and `k = 2`, the answer is `2` even though both walks end at the same destination. A DP table can accumulate those separate two-edge completions without exploring every recursive branch repeatedly.

## Reference Implementations

<ExamplePair>

```js
function countWalksWithExactlyKEdges(graph, source, destination, edgeCount) {
  const size = graph.length;
  const dp = Array.from({ length: edgeCount + 1 }, () =>
    Array.from({ length: size }, () => new Array(size).fill(0)),
  );

  for (let node = 0; node < size; node += 1) {
    dp[0][node][node] = 1;
  }

  for (let used = 1; used <= edgeCount; used += 1) {
    for (let from = 0; from < size; from += 1) {
      for (let to = 0; to < size; to += 1) {
        let total = 0;
        for (let next = 0; next < size; next += 1) {
          if (graph[from][next] > 0) {
            total += graph[from][next] * dp[used - 1][next][to];
          }
        }
        dp[used][from][to] = total;
      }
    }
  }

  return dp[edgeCount][source][destination];
}
```

## Follow-up Questions

- Reduce the memory usage by keeping only the previous edge-count layer.
- Explain the difference between counting walks and counting simple paths.
- Rework the solution with matrix exponentiation for larger `k`.

## Common Pitfalls

- Treating repeated vertices as invalid when the problem counts walks rather than simple paths.
- Forgetting the base case that a zero-edge walk exists from a node to itself.
- Mixing exact length with shortest path logic.
