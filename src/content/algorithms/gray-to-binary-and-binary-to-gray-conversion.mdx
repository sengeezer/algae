## Why It Surfaces

This problem comes up when the interviewer wants more than Gray-code generation. The key distinction is that generating Gray code is a direct formula, but decoding it back requires folding XOR information from the most significant side down through the remaining bits.

## Worked Example

For `n = 5`, binary-to-Gray returns `7` because `5 ^ (5 >> 1)` is `101 ^ 010 = 111`. Treating the same input as Gray code, Gray-to-binary returns `6` after cumulatively XORing the shifted suffixes.

## Reference Implementations

<ExamplePair>

```js
function grayToBinaryAndBinaryToGrayConversion(value) {
  if (!Number.isInteger(value) || value < 0 || value > 0xffffffff) {
    throw new RangeError('Value must be an unsigned 32-bit integer');
  }

  const number = value >>> 0;
  const binaryToGray = (number ^ (number >>> 1)) >>> 0;
  let grayToBinary = number;
  let shifted = number >>> 1;

  while (shifted !== 0) {
    grayToBinary ^= shifted;
    shifted >>>= 1;
  }

  return {
    binaryToGray,
    grayToBinary: grayToBinary >>> 0,
  };
}
```

## Common Pitfalls

- Assuming Gray-to-binary is another one-step XOR like binary-to-Gray.
- Confusing the generated Gray-code value with a list of Gray codes.
- Forgetting that JavaScript bitwise operators coerce values into 32-bit integers.
