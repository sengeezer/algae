## Why It Surfaces

Run-length encoding is a good first compression algorithm because the rule is obvious and reversible. It forces you to reason carefully about consecutive runs, formatting choices, and whether the encoded representation is actually smaller.

## Worked Example

`AAAABBBCCDAA` becomes `4A3B2C1D2A`. Each maximal run turns into one count-and-symbol pair, and decoding expands each pair back into repeated characters.

## Reference Implementations

<ExamplePair>

```js
function encodeRLE(text) {
  if (text.length === 0) {
    return '';
  }

  let encoded = '';
  let count = 1;

  for (let index = 1; index <= text.length; index += 1) {
    if (text[index] === text[index - 1]) {
      count += 1;
    } else {
      encoded += `${count}${text[index - 1]}`;
      count = 1;
    }
  }

  return encoded;
}

function decodeRLE(text) {
  let decoded = '';
  let count = '';

  for (const char of text) {
    if (char >= '0' && char <= '9') {
      count += char;
    } else {
      decoded += char.repeat(Number(count));
      count = '';
    }
  }

  return decoded;
}
```

## Follow-up Questions

- Explain when RLE shrinks data and when it makes it larger.
- Support inputs that may themselves contain digits safely.
- Extend the routine to arrays of repeated values instead of strings.

## Common Pitfalls

- Forgetting to flush the final run at the end of the scan.
- Assuming RLE always compresses the data.
- Using a decoder format that becomes ambiguous when raw symbols can be digits.
