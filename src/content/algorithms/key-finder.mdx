## Why It Surfaces

Key Finder is a brute-force search problem with a scoring layer on top. The key space is tiny, so the real decision is how to rank candidate decryptions without introducing noisy side effects or environment-specific behavior.

## Worked Example

Suppose a Caesar-encrypted sentence decodes into natural-looking English only when shifted by `7`. Trying all 26 shifts and counting common markers like `" the "`, `" and "`, or `" is "` will usually make that shift score highest.

<ExamplePair>

```js
function keyFinder(cipherText) {
  if (typeof cipherText !== 'string') {
    throw new TypeError('Cipher text must be a string');
  }

  const markers = [' the ', ' and ', ' is ', ' of ', ' to ', ' be '];

  function decodeWithShift(text, shift) {
    const normalized = ((-shift % 26) + 26) % 26;

    return text.replace(/[a-z]/gi, (char) => {
      const base = /[A-Z]/.test(char) ? 65 : 97;
      const index = char.charCodeAt(0) - base;
      return String.fromCharCode(base + ((index + normalized) % 26));
    });
  }

  let bestShift = 0;
  let bestScore = -1;

  for (let shift = 0; shift < 26; shift += 1) {
    const decoded = ` ${decodeWithShift(cipherText, shift).toLowerCase()} `;
    let score = 0;

    for (const marker of markers) {
      if (decoded.includes(marker)) {
        score += 1;
      }
    }

    if (score > bestScore) {
      bestScore = score;
      bestShift = shift;
    }
  }

  return bestShift;
}
```

## Follow-up Questions

- Return the best decoded plaintext along with the key.
- Use character-frequency scoring instead of a small marker dictionary.
- Explain why this brute-force approach works for Caesar but not for large-key ciphers.

## Common Pitfalls

- Adding environment-specific side effects like DOM writes into a pure search routine.
- Scoring only one hard-coded token and making the heuristic brittle.
- Forgetting that Caesar decryption just tries every normalized shift.
