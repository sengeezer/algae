## Why It Surfaces

Tree DP questions stop feeling mysterious once the node state is explicit. Unlike graph DP, the parent-child direction removes cycles, so the main job is deciding what information each subtree should return upward.

## Worked Example

In the article's sample tree, the maximum root-to-leaf sum is `22`. The best path is the root plus the child branch that eventually reaches the leaf with the highest accumulated total, so the tree can be solved from leaves back toward the root.

## Reference Implementations

<ExamplePair>

```js
function introductionToDynamicProgrammingOnTrees(root) {
  if (root === null) {
    return 0;
  }

  const memo = new Map();

  function solve(node) {
    if (node === null) {
      return Number.NEGATIVE_INFINITY;
    }

    const cached = memo.get(node);
    if (cached !== undefined) {
      return cached;
    }

    if (node.left === null && node.right === null) {
      memo.set(node, node.value);
      return node.value;
    }

    const answer = node.value + Math.max(solve(node.left), solve(node.right));
    memo.set(node, answer);
    return answer;
  }

  return solve(root);
}
```

## Interview Notes

- Define exactly what each node returns before writing transitions.
- Mention that postorder traversal is the usual bottom-up execution order.
- Call out rerooting when the interviewer wants answers for every possible root instead of one fixed root.

## Common Pitfalls

- Recomputing the same subtree answers instead of caching or combining them once.
- Forgetting to orient the tree by parent and child, which can create accidental cycles.
- Treating tree DP like array DP without defining what the node state actually means.
