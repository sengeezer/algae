## Why It Surfaces

This problem tests whether you know Euclid's algorithm deeply enough to re-express it without remainder arithmetic. The replacement is Stein's algorithm: strip common factors of two, keep values odd when possible, and use subtraction to keep shrinking the pair.

## Worked Example

For `48` and `18`, both numbers share one factor of `2`, leaving `24` and `9`. Repeating the shift-and-subtract process eventually reaches `3`, and restoring the shared factor gives a gcd of `6`.

```js
function euclidsAlgorithmWhenPercentAndDivisionOperationsAreCostly(left, right) {
  if (!Number.isSafeInteger(left) || !Number.isSafeInteger(right)) {
    throw new TypeError('Inputs must be safe integers');
  }

  let a = BigInt(Math.abs(left));
  let b = BigInt(Math.abs(right));

  if (a === 0n) {
    return Number(b);
  }

  if (b === 0n) {
    return Number(a);
  }

  let sharedTwos = 0n;
  while (((a | b) & 1n) === 0n) {
    a >>= 1n;
    b >>= 1n;
    sharedTwos += 1n;
  }

  while ((a & 1n) === 0n) {
    a >>= 1n;
  }

  while (b !== 0n) {
    while ((b & 1n) === 0n) {
      b >>= 1n;
    }

    if (a > b) {
      [a, b] = [b, a];
    }

    b -= a;
  }

  return Number(a << sharedTwos);
}
```

## Follow-up Questions

- Compare the binary-gcd recurrence with the standard remainder-based Euclidean algorithm.
- Extend the routine to `BigInt` inputs end to end instead of converting back to `number`.
- Explain why removing common powers of two first simplifies the remaining work.

## Common Pitfalls

- Reintroducing `%` or `/` even though the constraint is the whole point of the problem.
- Forgetting to restore the common factor of two that was removed from both inputs.
- Handling even and odd cases inconsistently and getting stuck in a non-decreasing loop.
