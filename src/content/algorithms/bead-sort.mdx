## Why It Surfaces

Bead sort shows up when someone wants a sorting example that is more about modeling than comparison logic. Each number becomes a row of beads, and gravity does the work by letting each column settle downward.

## Worked Example

For `[5, 3, 1]`, create three rows where the first row has five beads, the second has three, and the third has one. After each column settles downward, the rows become `[1, 3, 5]` when you count the beads left in each row.

## Reference Implementations

<ExamplePair>

```js
function beadSort(values) {
  if (!Array.isArray(values) || values.some((value) => !Number.isInteger(value) || value < 0)) {
    throw new RangeError('Values must be an array of non-negative integers');
  }

  if (values.length < 2) {
    return [...values];
  }

  const maxValue = Math.max(...values);
  const grid = values.map((value) => Array.from({ length: maxValue }, (_, index) => index < value));

  for (let column = 0; column < maxValue; column += 1) {
    let beads = 0;

    for (let row = 0; row < grid.length; row += 1) {
      if (grid[row][column]) {
        beads += 1;
      }
    }

    for (let row = grid.length - 1; row >= 0; row -= 1) {
      grid[row][column] = beads > 0;
      if (beads > 0) {
        beads -= 1;
      }
    }
  }

  return grid.map((row) => row.reduce((count, bead) => count + (bead ? 1 : 0), 0));
}
```

## Follow-up Questions

- Explain why bead sort is limited to non-negative integers.
- Compare its grid cost with counting sort on the same input range.
- Adapt the representation to make the gravity step easier to visualize on a whiteboard.

## Common Pitfalls

- Claiming bead sort works naturally for arbitrary comparable data.
- Ignoring the dependence on the maximum value when describing complexity.
- Forgetting that the simulation needs one column pass per possible bead height.
