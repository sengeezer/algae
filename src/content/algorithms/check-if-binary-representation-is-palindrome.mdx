## Why It Surfaces

This question checks whether you can reason about the live bit pattern directly instead of treating binary as just another display format. The clean solution mirrors a string palindrome check, but it does so with shifts and masks.

## Worked Example

`9` is `1001` in binary, which reads the same from both ends, so the function returns `true`. `10` is `1010`, which is not symmetric, so the function returns `false`.

## Reference Implementations

<ExamplePair>

```js
function checkIfBinaryRepresentationIsPalindrome(value) {
  if (!Number.isInteger(value) || value < 0 || value > 0xffffffff) {
    throw new RangeError('Value must be an unsigned 32-bit integer');
  }

  const number = value >>> 0;

  if (number === 0) {
    return true;
  }

  let left = 0;
  let right = Math.floor(Math.log2(number));

  while (left < right) {
    const leftBit = (number >>> left) & 1;
    const rightBit = (number >>> right) & 1;

    if (leftBit !== rightBit) {
      return false;
    }

    left += 1;
    right -= 1;
  }

  return true;
}
```

## Follow-up Questions

- Implement the same check by reversing the live binary suffix and comparing it with the original number.
- Explain why leading zeros are not part of the palindrome check here.
- Adapt the idea to decimal digits or fixed-width bitstrings.

## Common Pitfalls

- Treating leading zeros as significant when the prompt refers to the usual binary representation.
- Comparing decimal digits instead of binary bits.
- Forgetting that `0` should count as a binary palindrome here.
- Letting JavaScript bitwise coercion silently truncate values outside the unsigned 32-bit range.
