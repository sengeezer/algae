## Why It Surfaces

Bleak-number questions are really about bounding the search. Once you notice that `countSetBits(x)` can never exceed the number of bits in `x`, the huge naive scan collapses to a very small suffix of candidates.

## Worked Example

`3` is not bleak because `2 + countSetBits(2)` is `2 + 1 = 3`. `4` is bleak because no smaller non-negative `x` satisfies `x + countSetBits(x) = 4`.

```js
function countSetBits(value) {
  let number = BigInt(value);
  let count = 0;

  while (number > 0n) {
    number &= number - 1n;
    count += 1;
  }

  return count;
}

function checkIfANumberIsABleak(value) {
  if (!Number.isSafeInteger(value) || value < 1) {
    throw new RangeError('Value must be a positive safe integer');
  }

  const searchWidth = Math.ceil(Math.log2(value));
  const lowerBound = Math.max(0, value - searchWidth);
  const target = BigInt(value);

  for (let candidate = BigInt(lowerBound); candidate < target; candidate += 1n) {
    if (candidate + BigInt(countSetBits(candidate)) === target) {
      return false;
    }
  }

  return true;
}
```

## Follow-up Questions

- Compare the bounded search with the naive `1..n-1` scan.
- Replace the custom popcount helper with a built-in when the language provides one.
- Characterize ranges that contain many bleak numbers versus very few.

## Common Pitfalls

- Checking every predecessor up to `n - 1` even after deriving the logarithmic bound.
- Forgetting that the condition depends on the popcount of the candidate, not of `n`.
- Mishandling small positive inputs like `1`, where the logarithmic search width becomes `0`.
