## Why It Surfaces

Rotation problems test whether you are thinking in fixed-width binary rather than in ordinary arithmetic. The key distinction is that no bit is lost; every bit that falls off one side must reappear on the other.

## Worked Example

For `28` and a rotation distance of `2`, the 32-bit left rotation is `112` and the right rotation is `7`, so the function returns `[112, 7]`.

## Reference Implementations

<ExamplePair>

```js
function rotateBitsOfANumber(value, distance) {
  if (!Number.isInteger(value) || value < 0 || value > 0xffffffff) {
    throw new RangeError('Value must be an unsigned 32-bit integer');
  }

  if (!Number.isInteger(distance) || distance < 0) {
    throw new RangeError('Distance must be a non-negative integer');
  }

  const width = 32;
  const amount = distance % width;
  const number = value >>> 0;

  if (amount === 0) {
    return [number, number];
  }

  const left = ((number << amount) | (number >>> (width - amount))) >>> 0;
  const right = ((number >>> amount) | (number << (width - amount))) >>> 0;

  return [left, right];
}
```

## Follow-up Questions

- Split the helper into dedicated `rotateLeft` and `rotateRight` functions.
- Explain why `distance % 32` is necessary for fixed-width rotation.
- Compare circular rotation with ordinary shifts that discard overflowed bits.

## Common Pitfalls

- Treating rotation as a plain shift and losing the wrapped bits.
- Forgetting to normalize the distance by the word size.
- Ignoring the unsigned 32-bit model and returning negative signed results accidentally.
