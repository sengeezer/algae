## Why It Surfaces

Integer-palindrome checks are a classic digit-manipulation prompt. The arithmetic version is a good test of whether you can peel digits off cleanly without falling back to strings.

## Worked Example

`12321` is a palindrome because reversing its digits still gives `12321`. `120` is not, because reversing its digits gives `21`.

## Reference Implementations

<ExamplePair>

```js
function isPalindromeIntegerNumber(value) {
  if (typeof value !== 'number' || !Number.isFinite(value)) {
    throw new TypeError('Expected a finite number');
  }

  if (!Number.isInteger(value) || value < 0) {
    return false;
  }

  let remaining = value;
  let reversed = 0;

  while (remaining > 0) {
    reversed = reversed * 10 + (remaining % 10);
    remaining = Math.floor(remaining / 10);
  }

  return value === reversed;
}
```

## Follow-up Questions

- Compare the arithmetic approach with a string-based implementation.
- Reverse only half the digits to reduce overflow risk in fixed-width integer languages.
- Extend the predicate to other bases besides decimal.

## Common Pitfalls

- Forgetting that negative integers are not palindromes under the decimal-sign convention.
- Returning `true` for non-integer numeric inputs.
- Dropping leading zeros during reversal and misreading what that implies for values like `120`.
