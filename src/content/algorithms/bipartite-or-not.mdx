## Why It Surfaces

Bipartite detection is the guardrail that often appears just before matching or two-group assignment problems. The interview move is simple: color each component with two colors, and the first edge whose endpoints demand the same color proves the graph is not bipartite.

## Worked Example

Start with any uncolored vertex and assign it color `0`. Its neighbors must take color `1`, their uncolored neighbors must take color `0`, and so on; if you ever reach an edge connecting two vertices with the same assigned color, you found an odd-cycle contradiction.

## Reference Implementations

<ExamplePair>

```js
function bipartiteOrNot(graph) {
  const colors = new Array(graph.length).fill(-1);

  for (let start = 0; start < graph.length; start += 1) {
    if (colors[start] !== -1) {
      continue;
    }

    const queue = [start];
    colors[start] = 0;

    for (let head = 0; head < queue.length; head += 1) {
      const node = queue[head];

      for (const neighbor of graph[node]) {
        if (colors[neighbor] === -1) {
          colors[neighbor] = 1 - colors[node];
          queue.push(neighbor);
          continue;
        }

        if (colors[neighbor] === colors[node]) {
          return false;
        }
      }
    }
  }

  return true;
}
```

## Follow-up Questions

- Return the actual two partitions, not just a boolean answer.
- Explain why odd cycles are exactly what break bipartiteness.
- Compare this validation step with the later matching algorithms that assume a bipartite input.

## Common Pitfalls

- Assuming a disconnected graph can be checked from only one starting node.
- Forgetting to color vertices when they are enqueued, which can duplicate work.
- Confusing bipartite detection with finding a maximum bipartite matching.
