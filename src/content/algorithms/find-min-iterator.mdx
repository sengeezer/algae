## Why It Surfaces

This version matters when the input is not already an array. It checks whether you can consume an iterator safely, handle the empty case, and optionally project each value before comparing.

## Worked Example

If an iterable yields `10`, `7`, `12`, and `4`, the running minimum ends at `4`. If a selector maps objects to scores, the helper returns the smallest score it sees.

## Reference Implementations

<ExamplePair>

```js
function findMinIterator(iterable, selector) {
  if (iterable == null || typeof iterable[Symbol.iterator] !== 'function') {
    throw new TypeError('Expected an iterable value');
  }

  if (selector !== undefined && typeof selector !== 'function') {
    throw new TypeError('Selector must be a function when provided');
  }

  const iterator = iterable[Symbol.iterator]();
  let current = iterator.next();

  if (current.done) {
    return undefined;
  }

  let min = selector ? selector(current.value) : current.value;
  current = iterator.next();

  while (!current.done) {
    const candidate = selector ? selector(current.value) : current.value;
    if (candidate < min) {
      min = candidate;
    }
    current = iterator.next();
  }

  return min;
}
```

## Follow-up Questions

- Return the original item associated with the minimum projected value instead of the projected value itself.
- Compare consuming an iterator lazily with spreading it into an array and using `Math.min`.
- Support custom comparators instead of only numeric or lexical ordering.

## Common Pitfalls

- Assuming the iterable is non-empty.
- Forgetting that the selector-based version returns the minimum projected value, not the original element.
- Materializing the entire iterable when a single streaming pass is enough.
