## Why It Surfaces

This problem looks like a subset-enumeration trap, but the minimum product is driven almost entirely by sign structure. The greedy reasoning is to keep as many magnitude-helpful numbers as possible while removing only the one negative that would flip an otherwise ideal negative product into a positive one.

## Worked Example

For `a = [-1, -1, -2, 4, 3]`, there are three negative values, so using all nonzero elements keeps the overall product negative. The subset product is `(-1) * (-1) * (-2) * 4 * 3 = -24`, which is already minimal.

<ExamplePair>

```js
function minimumProductSubsetOfAnArray(values) {
  if (values.length === 0) {
    return 0;
  }

  if (values.length === 1) {
    return values[0];
  }

  let negativeCount = 0;
  let zeroCount = 0;
  let maximumNegative = Number.NEGATIVE_INFINITY;
  let minimumPositive = Number.POSITIVE_INFINITY;
  let product = 1;

  for (const value of values) {
    if (value === 0) {
      zeroCount += 1;
      continue;
    }

    product *= value;

    if (value < 0) {
      negativeCount += 1;
      maximumNegative = Math.max(maximumNegative, value);
    } else {
      minimumPositive = Math.min(minimumPositive, value);
    }
  }

  if (zeroCount === values.length) {
    return 0;
  }

  if (negativeCount === 0) {
    return zeroCount > 0 ? 0 : minimumPositive;
  }

  if (negativeCount % 2 === 0) {
    return product / maximumNegative;
  }

  return product;
}
```

## Follow-up Questions

- Return one actual subset, not only the minimum product value.
- Explain why removing the negative closest to zero is optimal when the negative count is even.
- Adapt the reasoning if overflow means you cannot multiply all values directly.

## Common Pitfalls

- Forgetting that the best answer can be a single positive or zero when no negative product is available.
- Removing the most negative value instead of the negative closest to zero.
- Multiplying zeros into the running product even though they should be handled as a separate case.
