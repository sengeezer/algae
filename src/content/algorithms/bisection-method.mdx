## Why It Surfaces

Bisection is the numerical-analysis version of binary search. It matters because it gives you a predictable, safe root finder when you can bracket a sign change but do not want to rely on a derivative.

## Worked Example

For $f(x) = x^2 - 2$ on `[1, 2]`, the function values have opposite signs, so a root lies inside. Repeated midpoint tests converge toward $\sqrt{2}$.

```js
function bisectionMethod(left, right, func, iterations = 50) {
  if (![left, right, iterations].every((value) => typeof value === 'number' && Number.isFinite(value))) {
    throw new TypeError('Bounds and iteration count must be finite numbers');
  }

  if (!Number.isInteger(iterations) || iterations <= 0) {
    throw new RangeError('Iterations must be a positive integer');
  }

  if (typeof func !== 'function') {
    throw new TypeError('Expected a function to evaluate');
  }

  let a = left;
  let b = right;
  let fa = func(a);
  let fb = func(b);

  if (!Number.isFinite(fa) || !Number.isFinite(fb)) {
    throw new RangeError('Function must be finite at both interval endpoints');
  }

  if (fa * fb > 0) {
    throw new RangeError('Interval endpoints must bracket a root with opposite-signed values');
  }

  for (let step = 0; step < iterations; step += 1) {
    const midpoint = (a + b) / 2;
    const fm = func(midpoint);

    if (fm === 0) {
      return midpoint;
    }

    if (fa * fm <= 0) {
      b = midpoint;
      fb = fm;
    } else {
      a = midpoint;
      fa = fm;
    }
  }

  return (a + b) / 2;
}
```

## Follow-up Questions

- Replace the fixed iteration count with a tolerance on interval width or function value.
- Compare bisection with Newton's method in terms of safety and speed.
- Explain why the sign-change precondition matters.

## Common Pitfalls

- Starting from endpoints that do not actually bracket a root.
- Forgetting to validate that the function is defined at both bounds.
- Losing the sign-change invariant when choosing the next interval.
