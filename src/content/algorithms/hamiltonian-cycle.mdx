## Why It Surfaces

Hamiltonian cycle questions test whether you can build a global tour from tight local constraints. The hard part is that visiting every vertex exactly once gives you very little reusable structure, so the usual interview solution is careful backtracking with aggressive feasibility checks.

## Worked Example

If the current path is `[0, 2, 1]`, the next candidate vertex must be adjacent to `1` and must not already appear earlier in the path. Once the path length reaches `n`, one final edge back to `0` determines whether you found a cycle or just a dead end.

```js
function hamiltonianCycle(graph) {
  const nodeCount = graph.length;
  if (nodeCount === 0) {
    return [];
  }

  const path = new Array(nodeCount).fill(-1);
  path[0] = 0;

  function isSafe(vertex, position) {
    if (graph[path[position - 1]][vertex] === 0) {
      return false;
    }

    for (let index = 0; index < position; index += 1) {
      if (path[index] === vertex) {
        return false;
      }
    }

    return true;
  }

  function search(position) {
    if (position === nodeCount) {
      return graph[path[nodeCount - 1]][path[0]] === 1;
    }

    for (let vertex = 1; vertex < nodeCount; vertex += 1) {
      if (!isSafe(vertex, position)) {
        continue;
      }

      path[position] = vertex;
      if (search(position + 1)) {
        return true;
      }

      path[position] = -1;
    }

    return false;
  }

  return search(1) ? [...path, path[0]] : [];
}
```

## Follow-up Questions

- Compare Hamiltonian cycle with Eulerian path, where the constraints apply to edges instead of vertices.
- Add pruning heuristics such as choosing the next vertex by degree.
- Explain why adding weights turns the problem into TSP.

## Common Pitfalls

- Forgetting that a Hamiltonian cycle must return to the starting vertex.
- Confusing "visit every vertex once" with "use every edge once."
- Treating degree counts as a complete correctness test the way they are for Eulerian graphs.
