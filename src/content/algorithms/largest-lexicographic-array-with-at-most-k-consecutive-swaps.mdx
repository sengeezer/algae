## Why It Surfaces

Because lexicographic order is decided from left to right, any swap budget spent improving an earlier index dominates equal effort spent later. The greedy strategy is therefore to examine only the reachable window from the current index, pull the largest value in that window forward, and subtract exactly the swaps used before moving on.

## Worked Example

For `arr = [3, 5, 4, 1, 2]` and `k = 3`, the best first move is to bring `5` to the front, leaving `[5, 3, 4, 1, 2]` with `k = 2`. Then bring `4` to index `1`, and finally bring `2` to index `3`, producing `[5, 4, 3, 2, 1]`.

## Reference Implementations

<ExamplePair>

```js
function largestLexicographicArrayWithAtMostKConsecutiveSwaps(values, swaps) {
  const result = [...values];

  for (let index = 0; index < result.length - 1 && swaps > 0; index += 1) {
    let bestIndex = index;
    const farthestIndex = Math.min(result.length - 1, index + swaps);

    for (let candidate = index + 1; candidate <= farthestIndex; candidate += 1) {
      if (result[candidate] > result[bestIndex]) {
        bestIndex = candidate;
      }
    }

    const bestValue = result[bestIndex];

    for (let move = bestIndex; move > index; move -= 1) {
      result[move] = result[move - 1];
    }

    result[index] = bestValue;
    swaps -= bestIndex - index;
  }

  return result;
}
```

## Follow-up Questions

- Prove why the earliest improvable position should always consume the next unit of budget.
- Return only the lexicographically largest prefix of length `m` if the rest of the array does not matter.
- Compare this greedy approach with a Fenwick-tree-assisted implementation that tracks shifts faster.

## Common Pitfalls

- Searching beyond the reachable window defined by the remaining swap budget.
- Spending extra swaps on a later equal maximum instead of the earliest one.
- Forgetting that moving a value left requires shifting the intervening elements right.
