## Why It Surfaces

Abbreviation is a strong DP interview problem because the operations look small but the decisions interact across prefixes. Greedy capitalization or deletion is not enough because a choice that works locally can block the rest of the target string.

## Worked Example

If `source = "daBcd"` and `target = "ABC"`, you can delete `d`, capitalize `a` into `A`, keep `B`, capitalize `c` into `C`, and delete the trailing `d`. A DP table records that each of those prefix decisions remains feasible.

<Callout title="Constraint hinge" tone="warning">
  Uppercase letters in the source cannot be discarded. If one of them fails to match the next target character, that whole DP path dies immediately.
</Callout>

## Reference Implementations

<ExamplePair>

```js
function canAbbreviate(source, target) {
  const dp = Array.from({ length: source.length + 1 }, () => new Array(target.length + 1).fill(false));
  dp[0][0] = true;

  for (let sourceIndex = 0; sourceIndex < source.length; sourceIndex += 1) {
    for (let targetIndex = 0; targetIndex <= target.length; targetIndex += 1) {
      if (!dp[sourceIndex][targetIndex]) {
        continue;
      }

      const sourceChar = source[sourceIndex];

      if (sourceChar === sourceChar.toLowerCase()) {
        dp[sourceIndex + 1][targetIndex] = true;
      }

      if (
        targetIndex < target.length
        && sourceChar.toUpperCase() === target[targetIndex]
      ) {
        dp[sourceIndex + 1][targetIndex + 1] = true;
      }
    }
  }

  return dp[source.length][target.length];
}
```

## Follow-up Questions

- Reduce the table to one row if only feasibility is needed.
- Return one valid sequence of operations instead of a boolean answer.
- Compare this with edit distance and subsequence matching.

## Common Pitfalls

- Deleting uppercase letters even though the rules forbid it.
- Using greedy capitalization instead of prefix DP.
- Forgetting that lowercase letters can either match or disappear.
